3. Description de la conception

3.1 Architecture générale

Le projet est structuré en deux parties distinctes :

  • Frontend : application Vue.js 3 avec Vite, gère l’interface utilisateur, l’authentification, les pages de profil, les posts, le matching et le chat.
  • Backend : application Python FastAPI, fournit une API REST pour l’authentification, la gestion des utilisateurs, la création de posts, le matching et la messagerie.

Le frontend communique avec le backend via des requêtes HTTP JSON. Le backend utilise SQLAlchemy pour accéder à la base de données PostgreSQL.

3.2 Structure du frontend

  • Routes principales : /, /login, /register, /dashboard, /profile, /posts, /matches, /chat.
  • Gestion de l’accès aux pages protégées avec un guard qui vérifie la présence d’un token dans localStorage.
  • Utilisation de composants Vue pour organiser les vues et la navigation.

3.3 Structure du backend

  • main.py : point d’entrée de l’API, configuration CORS et inclusion des routers.
  • config.py : lecture des variables d’environnement pour la base de données, Supabase, JWT.
  • database.py : création du moteur SQLAlchemy et du client Supabase pour la gestion du temps réel.
  • Dossiers principaux : models/, routers/, schemas/, services/.

3.4 Description du modèle de données

Le schéma relationnel de la base de données est conçu autour d’un réseau de mentorat :

TableDescription
usersStocke les informations des utilisateurs : prénom, nom, email, téléphone, mot de passe haché, filière, niveau, bio, photo.
skillsListe des compétences disponibles dans l’application.
user_skillsAssociation entre utilisateur et compétence avec niveau : strong pour mentor, weak pour besoin d’aide.
user_availabilitiesCréneaux horaires habituels d’un utilisateur.
mentorship_postsOffres ou demandes de mentorat publiées par les utilisateurs.
post_availabilitiesDisponibilités associées à chaque post.
matchesCorrélations mentor-mentoré calculées par l’algorithme de matching.
conversationsDiscussion associée à un match accepté.
messagesMessages échangés dans une conversation.

3.5 Fonctionnement du matching

L’algorithme de matching se base sur trois critères :

  • Compétence : le mentor doit maîtriser la compétence que le mentoré recherche.
  • Disponibilités : comparaison des créneaux horaires des deux utilisateurs.
  • Filière : mêmes filières ou filières proches augmentent le score.

Le score est calculé sur 100 points :

  • 40 points pour la compé- tence.
  • 30 points pour les disponibilités communes.
  • 30 points pour la proximité de filière.

4. Instructions de déploiement

4.1 Pré-requis

  • Node.js et npm installés pour le frontend.
  • Python 3.11+ et un environnement virtuel pour le backend.
  • Une base PostgreSQL ou Supabase configurée.
  • Un fichier .env contenant les variables de connexion.

4.2 Déploiement frontend

cd IFRI_MentorLink_Front
npm install
npm run dev

Le frontend se lance normalement sur http://localhost:5173.

4.3 Déploiement backend

cd IFRI_MentorLink-backend
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
uvicorn main:app --reload

Le backend est accessible par défaut sur http://localhost:8000. La route racine / retourne un message de fonctionnement, et la documentation Swagger est disponible via /docs.

4.4 Configuration de la base de données

La connexion est gérée dans IFRI_MentorLink-backend/config.py. Les variables attendues sont :

  • DATABASE_URL
  • SECRET_KEY
  • ALGORITHM
  • ACCESS_TOKEN_EXPIRE_MINUTES
  • SUPABASE_URL
  • SUPABASE_KEY

La base de données doit contenir les tables SQL correspondant aux modèles SQLAlchemy. Un script d’initialisation peut être généré à partir de init_db.py si nécessaire.

5. Manuel d’utilisation

5.1 Accès et inscription

  1. Ouvrir l’application frontend à http://localhost:5173.
  2. Cliquer sur « S’inscrire » puis remplir le formulaire avec prénom, nom, email ou téléphone, mot de passe, filière et niveau.
  3. Après inscription, l’utilisateur reçoit un token et est redirigé vers son espace.

5.2 Connexion

  1. Aller sur la page de connexion.
  2. Entrer l’email ou le téléphone et le mot de passe.
  3. Le token est stocké en local et permet d’accéder aux pages protégées.

5.3 Profil utilisateur

Depuis la page de profil, l’utilisateur peut consulter et mettre à jour ses informations personnelles, ses compétences et ses disponibilités.

5.4 Création et gestion des posts

L’utilisateur peut créer deux types de post :

  • Offer : proposer de l’aide sur une compétence.
  • Request : demander de l’aide sur une compétence.

Chaque post peut contenir une description, un mode (en ligne, présentiel ou mixte), et des créneaux de disponibilité.

5.5 Matching

La page de matching calcule automatiquement les correspondances entre utilisateurs en fonction de leurs compétences, disponibilités et filières. Les résultats sont affichés par score décroissant.

5.6 Messagerie

Lorsque deux utilisateurs sont associés par un match accepté, une conversation est créée. Les messages peuvent être envoyés et consultés dans l’onglet chat.

5.7 Bonnes pratiques

  • Vérifier que l’API backend est démarrée avant d’ouvrir le frontend.
  • Actualiser la page si le token de session expire et se reconnecter.
  • Archiver un post plutôt que le supprimer pour conserver l’historique.
